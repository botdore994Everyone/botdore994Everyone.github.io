1.Introduction to Web Application

1.1 Introduction

Web应用是运行在Web浏览器上的交互式应用，通常采用C/S架构来运行和处理交互。

通常有运行在客户端（浏览器）上的前端组件（即网站界面，或“用户看到的内容”）和运行在服务器端（后端服务器/数据库）上的其他后端组件（web应用程序源代码）。

Web Applications vs. Websites

Web 1.0是静态网站，无法实时进行交互，这类静态网站不包含函数，因此不会产生实时变化。

大多数网站运行的是Web应用程序（Web 2.0），能够根据用户交互呈现动态内容。传统网站和Web应用之间的区别在于：

  • 模块化
  • 任意显示比例运行

Web Applications vs. Native Operating System Applications

与 native OS不同，Web应用是跨平台的，可以在任何操作系统的浏览器中运行。Web应用及其功能在远程服务器上执行，因此不会占用用户的硬盘空间。

Web应用相比原生操作系统应用程序的一个优势是版本unity，所有访问Web应用的用户使用相同的版本和相同的Web应用程序，可以不断更新，而不需要向用户推送。

原生操作系统应用的优势在于操作速度以及利用原生操作系统数据库和本地硬件的能力。此外，原生应用通常更强大，因为它们与操作系统有更深入的集成，而不仅局限于浏览器功能。

近年来，混合式、渐进式的Web应用越来越普遍，它们利用现代框架使用原生操作系统的功能和资源运行web应用，比普通的web应用更快、更强大。

Web Application Distribution

有许多开源的Web应用被世界各地组织使用，它们可以被定制以满足企业需求。一些常见的开源Web应用程序包括：

  • WordPress
  • OpenCart
  • Joomla

还有闭源的Web应用，如：

  • Wix
  • Shopify
  • DotNetNuke

Security Risks of Web Applications

Web应用渗透测试是一门越来越重要的技能。任何希望保护面向互联网（和内部）web应用程序的组织都应该频繁地进行web应用程序测试，并在开发生命周期的每个阶段实施安全编码实践。

为了正确地渗透web应用程序，我们需要了解它们是如何工作的，它们是如何开发的，以及根据使用的技术，应用程序的每一层和组件存在什么样的风险。

OWASP Web安全测试指南是目前最流行、使用最广泛的Web应用程序测试方法之一。

Attacking Web Applications

Web应用，从简单的静态网站到由CMS支持的博客（如WordPress），再到具有注册/登录功能的复杂应用，几乎每家公司都有暴露在互联网的Web应用。

现在，通过已知的公开漏洞（如服务漏洞或Windows RCE漏洞）直接对外部主机攻击的情况并不常见。

Web应用的动态特性，意味着一段代码的更改可能引入一系列漏洞。这些漏洞可以组成漏洞链，以获取对敏感数据的访问，或在Web服务器或其它主机（如数据库服务器）上的远程代码执行。

在Active Directory 进行身份验证的Web应用程序上经常出现SQL注入漏洞。虽然不能提取密码，但是可以获取用户名或邮箱进行密码喷洒攻击。

1.2 Web Application Layout

Web应用布局有许多不同的层级组成，可以划分为以下三个主要类比额：

Category 类别 Description 描述
Web Application Infrastructure 描述web应用程序按预期运行所需的组件（如数据库）的结构。由于web应用程序可以设置为在单独的服务器上运行，因此必须知道它需要访问哪个数据库服务器。
Web Application Components 构成Web应用的组件，指所有与Web应用交互的组件。通常分为三个区域： UI/UX 组件, Client 组件，和 Server 组件。
Web Application Architecture 体系结构包含各种web应用程序组件之间的关系。

Web Application Infrastructure

Web应用可以使用多种不同的基础设施架构部署方式，这些方式被称为模型。最常见的模型可以分为以下四种类型：

  • Client-Server
  • One Server
  • Many Servers - One Database
  • Many Servers - Many Databases

Client-Server

C/S架构是Web应用最长采用的模型。在这种模型中，Web应用有两类组件，一类是前端组件，通常在客户端（浏览器）解释和执行；一类是后端组件，通常由托管的服务器编译、解释、执行。

  1. 用户访问URL，服务器加载Web应用的主界面
  2. 用户点击某个特定功能，浏览器向服务器发起一个 HTTP web request
  3. 服务器解析请求，并执行相应的操作
  4. 当服务器处理完毕后，会将结果返回给客户端浏览器（响应）
  5. 浏览器解析response并显示

One Server

整个Web应用程序，甚至几个Web应用程序及其组件（如数据库）托管在一台服务器上。虽然简单易行，但也是风险最大的设计，任何托管在该服务器上的Web应用存在漏洞，所有Web应用的数据都将受到威胁。
